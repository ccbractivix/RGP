Feasibility Assessment: Standalone Digital Signage System with Theater Integration
✅ The Direction is Sound
Building it as a standalone system is the right architectural call. What "Standalone" Means Practically
You'd essentially be building a small digital signage CMS — think of it like a lightweight version of Screenly, but self-hosted and owned by you. It has two parts:
1. The player — A browser-based display (/tv) that polls for its playlist and renders slides full-screen in a loop.

The theater integration point is a single, clean API call: the theater engine publishes its schedule data to a URL, and the signage system has a "Theater Schedule" slide type that knows how to fetch and render it. That's the only coupling — one direction, one endpoint.

The Multi-Display Question — Biggest Design Decision
"Different content around the property" means each display needs its own channel or playlist. A lobby screen shows the schedule + welcome message. The pool area shows activities + weather. The game room shows something else.
This requires a display identity model: each TV/player gets a unique ID (set once in the browser), and the signage server knows which playlist to serve to which ID. This is entirely doable but is the single most important design decision to get right upfront, because it shapes everything else.
Two approaches:
* Channel-based (simpler): You define named channels ("Lobby," "Pool," "Theater Foyer"). Each display is assigned a channel. Slides belong to channels. Easy to manage.
* Per-display playlists (more flexible): Every display has its own independent playlist. More powerful, more admin overhead.
For a property your size, channel-based is almost certainly the right fit — you likely have 3-5 logical zones, not 20 unique displays.

Suggestions Before You Commit
1. Define your channels first. Walk the property and list every location where you'd put a screen and what it would show. That list drives the whole design.
2. Decide on the theater integration contract now. The theater engine should expose a clean, versioned JSON endpoint (it basically already does via the Google Apps Script URL). The signage system should treat it as an external feed — never directly reading the theater spreadsheet.
3. Start with 2 slide types, not 6. Build the infrastructure (player loop, admin, channel system, display registration) with just "Announcement" and "Theater Schedule" slides first. Add slide types incrementally.
4. The admin doesn't have to be fancy. A simple form-based admin (add slide, set duration, pick channel, enable/disable) is 80% as useful as a drag-and-drop builder and takes 20% of the time. You can add polish later.

Bottom Line
Fully feasible. The right call architecturally. The main work is the admin UI and the channel/display model — the player itself is the easy part. Starting with a channel-based model and a dedicated signage Apps Script backend keeps you on zero-cost infrastructure while giving you a real, property-wide system.
